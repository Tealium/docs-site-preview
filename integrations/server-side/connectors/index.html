SERVER SIDE

Connectors

Connectors distribute customer data in real-time to any destination.

Summary

A connector is an integration between Tealium and a vendor for transmitting data. Connectors enable clients to send data from Tealium servers directly to vendors with a simplified user workflow in the Tealium interface. Learn more about connectors.

Tealium builds and provides API integrations for each connector. Clients may send data with two distinct scopes: event-level data and visitor-level data. Both scopes align with Tealium EventStream and Tealium AudienceStream. Users select a source [an event feed in EventStream, or visitor profile in AudienceStream], and the data is mapped to the available variables on the vendor side.

Connectors provide several different methods of data transfer, referred to as connector actions. The following are a few common connector actions for various vendor categories:

  • Display Ads
    Add or remove from an advertising segment
  • Email
    Add or remove from a list, trigger email, and add relational data for subscriber
  • Customer Relationship Management
    Create lead or contact, convert lead, and upsert [insert or update] record
  • Mobile
    Send SMS or a push notification, track an app install, and send in-app events

Benefits

Benefits of Tealium connector integrations include:

  • Since Tealium provides API integrations for each connector, clients do not need to manually build and maintain integrations to their vendors.
  • A single connector may have multiple connector actions.

Example

With the Facebook Audiences Connector for Facebook Ads users, Tealium clients are able to manage their custom audiences in real-time, based on user behavior collected from one or more data sources. This connector provides 3 actions:

  1. Add user to Custom Audience
  2. Remove user from Custom Audience
  3. Opt-out user from All Custom Audiences

For each action, Tealium displays sections for mapping client data to the required and optional fields for Facebook Custom Audiences.

Authentication

In the case where connectors require authentication, Tealium creates and manages a developer app for clients to act on their behalf (managed connectors). In other cases, Tealium requires the end-user to obtain necessary API access and to authenticate using their own credentials (unmanaged connectors).

For both managed and unmanaged connector credentials, Tealium supports the following authentication types:

  • Basic Auth (username and password)
  • OAuth 2.0 flows
  • JSON Web Tokens (JWT)

If your API places limits on the number of concurrent tokens, Tealium customers may encounter issues since we are globally distributed across dozens of servers at any given time.

Best Practices

Asset Creation

Asset creation is the ability to create resources for use with a connector without leaving the Tealium Customer Data Hub (CDH). For example, in the Facebook Audiences Connector example above, users are able to create a new Facebook custom audience and then select that audience as the target for a new connector action.

This optional feature is provided for all vendors that support API methods to create resources closely related to a supported connector action. Some common examples of create methods include:

  • Display Ads
    Create advertiser segment or audience
  • Email
    Create subscriber list or contact list

Dynamic Dropdowns for Action Configuration

Tealium assists users configuring their data export from Tealium to a technology partner by populating the mapping screens with relevant information the user is accustomed to seeing in their downstream platform.

For example, in the Facebook Audiences Connector example above, Tealium makes calls to GET lists of the user’s custom audiences from the Facebook Ads platform. This permits the user to browse their external resources with familiar names and helpful metadata to ensure they are selecting the right resources.

Dropdowns for action configurations may sometimes need to be executed as sequential calls. Fore example, when getting a Salesforce CRM object’s primary key field(s), we must first retrieve the list of CRM objects.

Record Upsertion by Default

For API methods that create or update records in a vendor platform, Tealium prefers native support for upsertion in a single request. This will mitigate unnecessary subsequent API calls. For example, before creating or updating a record, an API user must first verify whether a record exists. If the record doesn’t exist, the next API call creates a new record. If the record did exist, the next API call updates an existing record.

Batching & Rate Limiting

While Tealium’s goal is to provide real-time performance, some vendors prefer data be sent in small batches in order to manage resource costs. This is especially true when dealing with large volumes of data.

Tealium recommends vendors to rate limit individual clients using their APIs. This prevents a small group of high-volume users from impacting many clients.

Tealium connectors conform to the vendor’s desired thresholds are for the API methods used. The following are rate limits examples:

  • 50 events per request with a maximum of 10 requests per second = 500 requests/second
  • 10,000 events per minute with a maximum 10KB per request = ~167 requests/second

Limitations

Tealium connectors do not currently support the following:

  • File transfers
  • Vendor APIs that do not connect within 5000ms
  • Vendor APIs that do not respond to requests within 5000ms (10s for batch requests)
