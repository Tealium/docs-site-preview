ANDROID

Lifecycle Trackingモジュール

アプリのライフサイクルトラッキングイベントを提供します。

手動インストール

このモジュールはこちらでダウンロードできます：Tealium Android Lifecycle Trackingモジュール。

モジュールをアプリに追加するには、以下の手順に従います。

  1. tealium.lifecycle-1.x.x.aarを<PROJECT_ROOT>/<MODULE>/libsに追加します。

  2. Tealiumライブラリの依存関係をbuild.gradleファイルに追加します。

    dependencies {
implementation(name:'tealium.lifecycle-1.1.2', ext:'aar')
}

Mavenインストール

Tealium Mavenのレポジトリがプロジェクトのbuild.gradleファイルに追加されていることを確認します。

// Top-level build file where you can add configuration options common to all sub-projects/modules.
allprojects {
   repositories {
       jcenter()
       // Tealium Maven repo directive
       maven {
           url "https://maven.tealiumiq.com/android/releases/"
       }
   }
}

アプリケーションレベルのbuild.gradleファイルで、Tealiumライフサイクルモジュールを追加します。 現在のバージョンは1.1.2です。

dependencies {
   // only add if not already present
   implementation 'com.tealium:library:5.5.5'
   // add lifecycle to list of dependencies
   implementation 'com.tealium:lifecycle:1.1.2'
}

自動トラッキングされるイベント

このモジュールでは、以下のライフサイクルイベントをトラッキングします。

  • launch
  • sleep
  • wake

これらのイベントは、lifecycle_type変数に表示されます。

Tealiumのライフサイクルトラッキングでは、もっぱら使用状況をアクティビティのライフサイクルに関連付けます。

このライブラリはApplication.ActivityLifecycleCallbacks APIを利用し、onActivityResumedおよびonActivityPausedで条件を評価します。

有効化するには、以下を呼び出します。

Tealium.Config config = Tealium.Config.create(application, "YOUR_ACCOUNT", "YOUR_PROFILE", "ENVIRONMENT");
boolean isAutoTracking = true;

// Additional instance for Lifecycle tracking
LifeCycle.setupInstance("INSTANCE_NAME", config, isAutoTracking);
// Standard Tealium instance
Tealium.createInstance("INSTANCE_NAME", config);

launchイベント

最初のonActivityResumedで生成されて発生します。 onActivityResumedの前にonActivityPausedが呼び出されると（Activityライフサイクルの途中でライブラリが初期化された場合）モジュール作成の時間が、launchイベントの生成に使用されます。

詳細を見る

sleepイベント

onActivityPausedが呼び出され、onActivityResumedが呼び出されなかった5秒後に生成されます。 これは新しいビューが表示されなかったために、アプリケーションがバックグラウンドで動作していることを示します。

詳細を見る

wakeイベント

最後のonActivityPausedから5秒以上経過した場合に、onActivityResumedの最中に生成されます。 これはビューが変更されたからではなく、アプリケーションがフォアグラウンドで動作しているためにonActivityResumedが呼び出されたことを示します。

詳細を見る

手動でトラッキングされるイベント

特定のライフサイクルパラダイムおよび通常とは異なるライフサイクルトラッキングに使用できます。 ほとんどの実装には、この方法は推奨しません。 手動のライフサイクルトラッキングを許可するには、ライフサイクル初期化コードを更新し、isAutoTrackingをfalseに設定します。

launchイベントを手動でトラッキングするには、trackLaunchEventを呼び出します。

LifeCycle lifeCycle = LifeCycle.getInstance(TEALIUM_INSTANCE_ID);

if(lifeCycle != null) {

    // OPTIONAL
    Map<String, String> data = new HashMap<>();
    data.put("boot_time", "5ms");

    lifeCycle.trackLaunchEvent(data);
}

sleepイベントを手動でトラッキングするには、trackSleepEventを呼び出します。

LifeCycle lifeCycle = LifeCycle.getInstance(TEALIUM_INSTANCE_ID);

if(lifeCycle != null) {

    // OPTIONAL
    Map<String, String> data = new HashMap<>();
    data.put("foreground_time", "4000ms");

    lifeCycle.trackSleepEvent(data);
}

wakeイベントを手動でトラッキングするには、trackWakeEventを呼び出します。

LifeCycle lifeCycle = LifeCycle.getInstance(TEALIUM_INSTANCE_ID);

if(lifeCycle != null) {

    // OPTIONAL
    Map<String, String> data = new HashMap<>();
    data.put("background_time", "600000ms");

    lifeCycle.trackWakeEvent(data);
}

Tealiumおよび付随するLifeCycle Multitonは、Tealium iQの公開設定からリモートで終了できます。 インスタンスでメソッドを呼び出す場合は、Nullチェックを実行してください。

クラッシュ検知

wakeイベントに続いてlaunchイベントが発生する場合、またはその逆の場合も、アプリケーションが正常にsleepしなかったためクラッシュが発生したことを示します。 Tealiumライブラリのシャットダウンにより上記のような連続イベントが発生した場合は、クラッシュイベントとは見なされません。

クラッシュイベントが検知されると、launchまたはwakeイベントデータにlifecycle_diddetectcrash変数が追加されます。

{
    "lifecycle_type" : "launch",
    "lifecycle_diddetectcrash" : "true",
    //...
}

データレイヤー変数

このモジュールにより、リストに記載されたイベントのタイプ（launch、wake、sleep）に合わせて、以下の変数が追加されます。

変数 詳細 型 例 イベント
lifecycle_dayofweek_local コールが行われた現地曜日（1 =日曜日、2 =月曜日など） 数値 “13” すべて
lifecycle_dayssincelaunch 整数の増分で最初のlaunchから経過した日数 数値 “23” すべて
lifecycle_dayssinceupdate 整数の増分でアプリのバージョン更新が最後に検出されてから経過した日数 数値 “46” すべて
lifecycle_dayssincelastwake 整数の増分で最後にwakeが検出されてから経過した日数 数値 “1” すべて
lifecycle_diddetectcrash launchイベントがwakeイベントに続いた場合にのみ存在（先行するsleepイベントがないためクラッシュが推測される） ブーリアン “true” launch
lifecycle_firstlaunchdate launch/wakeが最初に検出されたときのGMTタイムスタンプ（ISO8601形式、協定世界時） 文字列 “2013-07-11T17:55:04Z”  すべて 
lifecycle_firstlaunchdate_MMDDYYYY  MM/DD/YYYY形式のGMTタイムスタンプ  文字列 “01/18/2012”  すべて
lifecycle_hourofday_local コールが行われた現地時間（24時間表記）  文字列 “true”  すべて
lifecycle_isfirstlaunch コールが、最初のlaunchコールである場合にのみ存在 文字列 “true” launch
lifecycle_isfirstlaunchupdate  コールが、更新検出後の最初のlaunchである場合にのみ存在 ブーリアン “true” launch
lifecycle_isfirstwakemonth コールが、月の最初のlaunch/wakeである場合にのみ存在 ブーリアン “true” launch、wake 
lifecycle_isfirstwaketoday コールが、1日の最初のlaunch/wakeである場合にのみ存在  ブーリアン  “true” launch、wake
lifecycle_launchcount 現バージョンのアプリのlaunchの総回数（最後の更新以降） 数値 “3”  すべて
lifecycle_secondsawake アプリが最後のwake/launchからwake状態にあった期間の総秒数。 lifecycle_type:sleep、またはlifecycle_type:terminateコールでのみ送信 数値 “30” すべて 
lifecycle_priorsecondsawake  アプリが最後のlaunchからwake状態にあった期間の総秒数。 これまでのすべてのwakeを総計。 lifecycle_type: launchコールでのみ送信。  数値  “126”  すべて
lifecycle_sleepcount アプリがsleepになった総回数（更新時にリセット）  数値  “5”   すべて
lifecycle_totalcrashcount インストール後のクラッシュの総回数（アプリが削除された場合のみリセット）  数値  “21”   すべて
lifecycle_totallaunchcount インストール後のlaunchの総回数（アプリが削除された場合のみリセット）  数値  “3”   すべて
lifecycle_totalsecondsawake  アプリがインストール後にwake/アクティブな状態にあった期間の総秒数（アプリが削除された場合のみリセット）  数値  “36”  すべて
lifecycle_totalsleepcount  アプリがインストール後にバックグラウンドに移行した総回数（アプリが削除された場合のみリセット）   数値  ”400”  すべて
lifecycle_totalwakecount  インストール後のlaunchとwake両方の総回数（アプリが削除された場合のみリセット）  数値   ”563”  すべて
lifecycle_type   ライフサイクルコールの種類 文字列 “initial”、”launch”、”wake”、”sleep”、”terminate”  すべて
lifecycle_updatelaunchdate  バージョン更新検出後、最初のwake/launchのGMTタイムスタンプ  文字列  “2014-09-08T18:10:01Z”  すべて
lifecycle_wakecount  現バージョンのアプリのlaunchとwake両方の総数（更新時にリセット）  数値   ”29”  すべて
